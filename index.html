1. ภาพรวมโมดูล

ระบบส่วนนี้ไม่ได้เป็นแค่ “หน้าบันทึกค่าใช้จ่าย” แต่เป็นวงจรครบตั้งแต่บันทึกค่าใช้จ่ายจริง, ขอเบิกเงินล่วงหน้า, อนุมัติหลายขั้น, โอนเงิน, เคลียร์เงินจริง, และปรับปรุงค่าใช้จ่ายหลังโอน

Module A Job Actual Expense
บันทึกค่าใช้จ่ายจริงของงาน
Module B Cash Advance
ขอเบิกเงินล่วงหน้าแบบ 1 Job หลายรายการ
Module C Settlement
เคลียร์เงินจริงหลังโอนเงินแล้ว
Module D Expense Adjustment
แก้ actual expense หลังฐานรายการถูก lock

ภาพใหญ่ของ flow ทั้งระบบ

1. สร้าง / ดู Job ผู้ใช้บันทึกข้อมูลงานและค่าใช้จ่ายจริงที่เกี่ยวข้อง
→
2. ขอเบิก Cash Advance 1 ใบเบิกผูกกับ 1 Job แต่มีได้หลาย request line
→
3. อนุมัติหลายขั้น CS Supervisor → Finance Verify → Final Approval → Accounting
4. โอนเงินจริง บัญชีบันทึกการโอน และแนบหลักฐานการโอน
→
5. Settlement เคลียร์จริงแบบ line-level, กระจายยอด, top-up, return, adjustment
→
6. ปิดรายการ เมื่อส่วนต่างเป็นศูนย์และเอกสารครบ สถานะเปลี่ยนเป็น Settled
สรุปแบบสั้นที่สุด: ค่าใช้จ่ายจริงใช้สำหรับ “บันทึกสิ่งที่เกิดขึ้นแล้ว”, Cash Advance ใช้สำหรับ “ขอเงินก่อนใช้จริง”, Settlement ใช้สำหรับ “เคลียร์เงินที่ขอไปให้ตรงกับสิ่งที่เกิดขึ้นจริง”, และ Expense Adjustment ใช้เมื่อ “ค่าใช้จ่ายจริงตัวเดิมถูก lock แล้วจะแก้ผ่านรายการปรับแทน”

2. บทบาทและสิทธิ์

ในเชิงระบบ สิทธิ์ถูกผูกกับ permission key ไม่ได้ผูกกับตำแหน่งงานชื่อเดียวตายตัว ดังนั้นผู้ใช้หนึ่งคนอาจถือหลายบทบาทได้

บทบาทเชิงธุรกิจ สิทธิ์หลักที่ใช้ ทำอะไรได้ หมายเหตุสำคัญ
ผู้บันทึกงาน / ผู้ขอเบิก transport.job create/edit + transport.cash_advance create/submit สร้าง Job, บันทึกค่าใช้จ่ายจริง, สร้างใบเบิกเงินล่วงหน้า ถ้าไม่มีสิทธิ์ดูแล Job จะเริ่มสร้าง Cash Advance ไม่ได้
ผู้ดูข้อมูล transport.cash_advance view, transport.expense_adjustment view ดูรายการ, ดูสถานะ, ดาวน์โหลดเอกสาร ดู Settlement ได้เมื่อรายการอยู่สถานะ Transferred หรือ Settled
หัวหน้า CS cash_advance cs_approve หรือ cash_advance approve
expense_adjustment cs_approve หรือ expense_adjustment approve		 อนุมัติขั้นหัวหน้า CS และส่งกลับ Draft Cash Advance ใช้สิทธิ์ role เป็นหลัก, Expense Adjustment ยังดู assigned user ด้วย
การเงินตรวจสอบ cash_advance finance_verify หรือ cash_advance approve ตรวจความพร้อมทางบัญชีของใบเบิกก่อนเข้าขั้นอนุมัติสุดท้าย ทุก request line ต้องมี Account Code ก่อนผ่านขั้นนี้
ผู้อนุมัติขั้นสุดท้าย cash_advance final_approve หรือ cash_advance approve อนุมัติ / ไม่อนุมัติรายบรรทัด, ส่งรายการต่อให้บัญชี ระบบบังคับให้มีอย่างน้อย 1 line ที่อนุมัติ
บัญชี cash_advance accounting_approve หรือ cash_advance approve บันทึกการโอนเงินจริงและแนบ slip หลังโอนแล้ว ใบเบิกถูก lock และต้องไปแก้ผ่าน Settlement เท่านั้น
ผู้เคลียร์เงิน cash_advance settle หรือ cash_advance approve Allocate ค่าใช้จ่ายจริง, บันทึก return / top-up / manual adjustment, ปิด settlement แก้ Settlement ได้เฉพาะสถานะ Transferred
ผู้จัดการการเงิน (Adjustment) expense_adjustment final_approve หรือ expense_adjustment approve อนุมัติ Expense Adjustment ขั้นสุดท้าย Expense Adjustment จะดู assigned user ถ้าระบุไว้
ผู้ดูแลระบบ สิทธิ์ตาม role ภายในระบบ ทำงานแทน role ต่าง ๆ ตามสิทธิ์ที่ได้รับ สำหรับ Expense Adjustment ถ้า roleId = 1 จะ bypass เงื่อนไข assigned user
จุดสำคัญมาก: Cash Advance ในปัจจุบัน “ใช้สิทธิ์ตาม role เป็นหลัก” แม้ว่าระบบจะเก็บผู้รับผิดชอบรายขั้นไว้เพื่อ routing และแสดงผล แต่ไม่ได้บังคับว่าต้องเป็น user ที่ถูก assign เท่านั้นจึงกดอนุมัติได้
ข้อยกเว้น: Expense Adjustment ยังใช้ตรรกะเข้มกว่า คือถ้าระบุ assigned user ไว้ ผู้ที่จะกดอนุมัติต้องเป็นคนนั้นจริง และผู้ขอรายการเองกดอนุมัติรายการตัวเองไม่ได้

3. หน้าจอหลักของระบบ

ถ้าจะสอนผู้ใช้ใหม่ แนะนำให้เริ่มจาก 5 หน้าจอนี้ก่อน

1. Job View

ใช้ดูรายละเอียดงาน, บันทึก Actual Expense, เลือกค่าใช้จ่ายที่จะใช้เป็นฐานในการสร้าง Cash Advance และดูว่ารายการใดถูก lock แล้ว

/transport/jobs/view/{jobId}

2. Cash Advance List

หน้ารายการใบเบิกทั้งหมด ใช้ค้นหา, กรองสถานะ, ดูคิวงาน และทำ Bulk Approval

/transport/cash-advances

3. Cash Advance Detail

ใช้ดูข้อมูลใบเบิก, request line summary, approval log, transfer slip, settlement summary และปุ่ม action รายขั้น

/transport/cash-advances/view/{id}

4. Settlement Workbench

ใช้เคลียร์เงินจริงหลังโอนเงินแล้ว โดยทำงานแบบ line-level และแนบเอกสารประกอบ

/transport/cash-advances/settlement/{id}

5. Expense Adjustment

ใช้ปรับ actual expense หลังรายการฐานถูก lock จากการมี cash advance ที่โอนแล้ว หรือมี adjustment ค้างอยู่

/transport/expense-adjustments

6. Expense Control Dashboard

workspace สำหรับฝ่ายคุมค่าใช้จ่าย ดูยอดวันนี้/สัปดาห์นี้, คิวโอน, คิวเคลียร์, open adjustment และเปิด settlement แบบเร็ว

/transport/expense-control

4. Actual Expense ที่ระดับ Job

Actual Expense คือ “ค่าใช้จ่ายจริงที่เกิดขึ้นกับงาน” และเป็นฐานข้อมูลสำคัญสำหรับทั้ง Cash Advance และ Settlement

สิ่งที่ทำได้

  • เพิ่มรายการค่าใช้จ่ายจริงในหน้า Job
  • แก้ไขรายการเดิม ก่อนที่ระบบจะ lock
  • ลบรายการ ก่อนที่ระบบจะ lock
  • ใช้รายการเหล่านี้เป็นรายการอ้างอิงตอนสร้าง Cash Advance

เมื่อไรที่ Actual Expense จะถูก lock

กรณีที่ 1: มี Cash Advance โอนแล้วใน Job เดียวกัน

ถ้างานนี้มี cash advance ที่อยู่หลังขั้นโอนแล้ว ระบบจะไม่ให้แก้ไขฐาน Actual Expense ตรง ๆ

กรณีที่ 2: มี Expense Adjustment ที่อนุมัติแล้ว

ถ้ารายการค่าใช้จ่ายนี้เคยถูกปรับผ่าน adjustment และได้รับอนุมัติแล้ว ฐานรายการเดิมจะถูก lock

กรณีที่ 3: มี Expense Adjustment ค้างอยู่

ถ้ามี adjustment แบบยังไม่จบ workflow ระบบก็ lock ฐานรายการไว้เช่นกัน เพื่อกันข้อมูลสวนทางกัน

ผลลัพธ์

เมื่อ lock แล้ว ต้องแก้ผ่าน Expense Adjustment หรือแก้ยอดใน Settlement ตามบริบท แทนการแก้รายการฐาน

หลักการใช้งาน: ถ้าระบบขึ้นว่า Actual Expense ถูก lock หลังโอนเงินหรือมี adjustment ค้างอยู่ อย่าพยายามแก้ฐานรายการเดิม ให้ใช้หน้า Expense Adjustment หรือ Settlement แทน

5. Cash Advance / ค่าใช้จ่ายล่วงหน้า

ปัจจุบันระบบรองรับ 1 ใบเบิก ต่อ 1 Job แต่ใน 1 ใบเบิกสามารถมี หลาย request line ได้ โดย 1 line หมายถึง “1 ประเภทค่าใช้จ่าย” ไม่ใช่ 1 ใบเสร็จ

โครงสร้างข้อมูลของใบเบิก

Header

มีเลขที่ใบเบิก, Job, วันที่ขอ, วันที่ต้องใช้เงิน, ผู้รับเงิน, หมายเหตุ และยอดรวมทั้งใบ

Request Line

แต่ละบรรทัดมี Cost Type, Account Code, Qty, Unit Price, Total, Description, Reference, Notes

Link กับ Job Expense

แต่ละ line ผูกกับ job expense ที่อยู่ใน Job เดียวกันได้ เพื่อระบุว่าเบิกเผื่อค่าใช้จ่ายตัวไหน

Validation ตอนสร้าง / แก้ไขใบเบิก

  • ต้องมีอย่างน้อย 1 request line
  • ทุก line ต้องเลือก Cost Type
  • Qty ต้องมากกว่า 0
  • Total Amount ต้องมากกว่า 0
  • ถ้าเลือก linked job expenses ต้องเลือกได้เฉพาะค่าใช้จ่ายจาก Job เดียวกัน
  • ระบบจะสรุปยอดรวมจากทุก line เป็นยอดใบเบิก

Flow การทำงานของ Cash Advance

Draft ผู้ขอบันทึกข้อมูลได้เต็มที่ แก้ไขหรือลบได้
→
Pending CS Supervisor รอหัวหน้า CS อนุมัติ หรือส่งกลับ Draft
→
Pending Finance Verify รอการเงินตรวจสอบ Account Code และความพร้อม
→
Pending Final Approval ผู้อนุมัติขั้นสุดท้ายอนุมัติหรือไม่อนุมัติราย line ได้
Pending Accounting บัญชีบันทึกการโอนเงินจริง
→
Transferred เงินถูกโอนแล้ว ใบเบิกถูก lock และไปต่อที่ Settlement
→
Settled เคลียร์เงินจริงและปิดส่วนต่างครบแล้ว

ตารางสถานะและ action ของ Cash Advance

สถานะ ใครทำงานต่อ Action หลัก หมายเหตุ
Draft ผู้ขอเบิก แก้ไข, ลบ, ส่งอนุมัติ ถ้าเลือกส่งอนุมัติ สถานะจะไป Pending CS Supervisor
Pending CS Supervisor หัวหน้า CS Approve, Return to Draft เป็นขั้นแรกของ approval
Pending Finance Verify การเงินตรวจสอบ Verify, Return to Draft ทุก line ต้องมี Account Code ก่อนผ่านขั้นนี้
Pending Final Approval ผู้อนุมัติขั้นสุดท้าย อนุมัติ / ไม่อนุมัติราย line, Reject ทั้งใบ ต้องมีอย่างน้อย 1 line ที่ approved จึงส่งต่อบัญชีได้
Pending Accounting บัญชี บันทึกโอนเงิน, Reject ทั้งใบ แนบ transfer slip และบันทึก transfer ref / transfer date
Transferred ผู้เคลียร์เงิน เปิด Settlement, เคลียร์เงินจริง หลังโอนแล้วห้ามแก้ใบเบิก ใช้ Settlement เท่านั้น
Settled ผู้ดูข้อมูล ดูสรุป, ดาวน์โหลดเอกสาร ปิด flow เรียบร้อยแล้ว
Rejected ผู้ขอเบิก แก้ไข, ส่งใหม่, ลบ ถือว่าเป็นรายการที่ไม่ผ่าน approval
Cancelled ผู้มีสิทธิ์แก้ไข แก้ไข, ลบ, ส่งใหม่ สถานะนี้มีอยู่ในตรรกะระบบ แม้ไม่ใช่ flow หลักที่ใช้งานทุกวัน

กติกา approval ของ Cash Advance

CS Supervisor

ตรวจความถูกต้องเบื้องต้นและส่งต่อให้การเงินตรวจสอบ

Finance Verify

ต้องมี Account Code ครบทุก line และต้อง route ผู้อนุมัติขั้นสุดท้ายได้ก่อนส่งต่อ

Final Approval แบบราย line

อนุมัติหรือ reject แต่ละ line ได้ ถ้า reject line ใดต้องใส่เหตุผลใน line นั้น

Approved Total

ยอดที่ส่งต่อให้ Accounting คือผลรวมของ line ที่ได้รับอนุมัติจริง ไม่ใช่ยอดเดิมทั้งใบเสมอไป

เกณฑ์ routing ผู้อนุมัติขั้นสุดท้าย

ยอดรวมคำขอ ระดับผู้อนุมัติ หลักการเลือก
ไม่เกิน 10,000 Manager ระบบเลือกจาก bucket ผู้มีสิทธิ์ final manager
มากกว่า 10,000 ถึง 50,000 Director ระบบเลือกจาก bucket ผู้มีสิทธิ์ final director
มากกว่า 50,000 Managing Director ระบบเลือกจาก bucket ผู้มีสิทธิ์ final MD

ข้อมูลที่ต้องครบก่อน Final Approval

  • Borrower Type ต้องเป็น driver หรือ freelance
  • ต้องเลือกผู้รับเงินจริงให้ตรงตามประเภท
  • Borrower Name Snapshot ต้องมี
  • Borrower Affiliation ต้องมี
  • Borrower Role ต้องมี
  • Advance Date ต้องมี

การแก้ไขหลังส่งอนุมัติ

หลักปัจจุบันของระบบ: ก่อนโอนเงิน ถ้ามีการแก้ข้อมูลสำคัญในช่วง Pending Final Approval หรือ Pending Accounting ระบบจะ restart approval กลับไปเริ่มที่ CS Supervisor ใหม่
หลังโอนเงินแล้ว: ห้ามแก้ใบเบิกโดยตรง ต้องไปปรับยอดและปิดส่วนต่างผ่าน Settlement Workbench เท่านั้น

Bulk Approval

  • มีในหน้ารายการ Cash Advance
  • รองรับ 3 ขั้น: CS Approval, Finance Verify, Final Approval
  • ใช้ note เดียวกันกับรายการที่เลือกทั้งหมดได้
  • Checkbox จะเปิดให้เลือกเฉพาะรายการที่อยู่ในสถานะที่ action นั้นทำได้
  • Bulk Final Approval จะอนุมัติทุก line ของแต่ละใบอัตโนมัติ ไม่ใช่หน้าสำหรับ reject บาง line

6. Settlement Workbench

Settlement ใช้หลังจากบัญชีโอนเงินจริงแล้วเท่านั้น เพื่อเคลียร์ว่าเงินที่เบิกไป “ใช้จริงเท่าไร, คืนเท่าไร, ต้องจ่ายเพิ่มเท่าไร”

เมื่อไรเข้า Settlement ได้

  • รายการต้องอยู่สถานะ Transferred หรือ Settled จึงเปิดดูได้
  • ถ้าจะ “แก้ไข / บันทึก / ปิด settlement” รายการต้องอยู่สถานะ Transferred
  • ผู้ใช้ต้องมีสิทธิ์ cash_advance settle หรือ cash_advance approve

รูปแบบ Settlement ปัจจุบัน

Line-level

ระบบปัจจุบันเคลียร์แบบราย request line ได้แล้ว แต่ละ line มี actual, allocated, variance, resolution, remaining ของตัวเอง

1 line แตกเป็นหลาย actual expense ได้

1 request line สามารถ allocate ไปหลายค่าใช้จ่ายจริงได้ ตราบใดที่อยู่ใน Job เดียวกันและผ่านกฎ matching

เอกสารแนบระดับบรรทัด

แต่ละ request line มีเอกสารแนบของตัวเองได้ สูงสุด 5 ไฟล์ต่อ line และระบบยังแสดง legacy document เก่าแยกไว้

Flow ของ Settlement

1. เปิด Workbench ระบบดึงทุก request line ที่ approved และยอด advance ของแต่ละ line
→
2. Allocate ค่าใช้จ่ายจริง เลือก actual expense ให้ตรง line นั้นและใส่ allocation amount
→
3. Resolve ส่วนต่าง บันทึก Top-up, Return หรือ Manual Adjustment ที่ระดับ line
4. แนบเอกสาร แนบ receipt / support document ใน line ที่เกี่ยวข้อง
→
5. Save Draft เก็บงานไว้ก่อน หากยังมีส่วนต่างค้างอยู่
→
6. Close Settlement ปิดได้เมื่อ remaining variance รวมเป็น 0 และเอกสารครบ

กติกา allocation ของ Settlement

  • allocate ได้เฉพาะ actual expense ที่อยู่ใน Job เดียวกัน
  • 1 actual expense ถูก allocate ได้กับ request line เดียวเท่านั้นใน settlement เดียวกัน
  • ระบบยอมให้ line ดึง expense ที่ “ตรง cost type เดียวกัน” หรือ “อยู่ใน planned linked expense list”
  • allocation amount ห้ามเกิน effective actual ของ expense นั้น

ความหมายของค่าที่ใช้ใน Settlement

ช่อง / ตัวเลข ความหมาย ใช้เมื่อไร
Actual Total ยอดใช้จริงรวมจาก actual expense ที่ allocate ไว้ ใช้ดูว่าค่าใช้จริงสูงหรือต่ำกว่าเงินที่เบิกไป
Allocated Total ยอด allocation ที่เลือกเข้ามาใน settlement ใช้ตรวจว่ากระจายยอดครบหรือยัง
Variance Actual - Advance บอกส่วนต่างก่อนแก้ด้วย return / top-up / adjustment
Top-up Amount จำนวนเงินที่ต้องจ่ายเพิ่มให้ line นั้น เมื่อ actual สูงกว่า advance
Return Amount จำนวนเงินที่ต้องโอนคืน เมื่อ advance สูงกว่า actual
Manual Settlement Adjustment การปรับ signed amount ด้วยมือ ใช้เฉพาะกรณีที่มีเหตุผลชัดเจนและต้องการปิดส่วนต่างแบบ manual

Settlement Reason

ตอนกด Close Settlement ระบบบังคับให้เลือกเหตุผลการปิดรายการ โดยตัวเลือกหลักคือ:

  • Normal settlement ปิดรายการตามปกติ
  • Job cancelled งานถูกยกเลิก
  • Unused advance returned มีเงินเหลือและโอนคืน
  • Other กรณีอื่น และต้องกรอก Reason Detail เพิ่ม

กติกาเอกสารแนบ

  • Settlement ต้องมีเอกสารแนบอย่างน้อย 1 ไฟล์ก่อน Save หรือ Close
  • ใน mode line-level ระบบรองรับเอกสารแนบราย request line
  • สูงสุด 5 ไฟล์ต่อ request line
  • เอกสารเก่าที่ไม่ผูก line จะถูกแสดงในส่วน Legacy / Unassigned Settlement Documents
  • ไฟล์ที่รองรับคือ PDF, JPG, JPEG, PNG, WEBP และแต่ละไฟล์ต้องไม่เกิน 10 MB
ความหมายทางธุรกิจ: การ “โอนคืน” ในระบบปัจจุบันไม่ได้แสดงแค่ตัวเลข แต่เป็น record จริงใน settlement จึงสามารถดูย้อนหลังได้จากสรุป settlement และจากเอกสารแนบที่ line นั้นเกี่ยวข้อง

7. Expense Adjustment

Expense Adjustment คือช่องทางแก้ actual expense หลังรายการฐานถูก lock แล้ว และใช้กับรายการที่มี cash advance โอนแล้วเป็นหลัก

เมื่อไรควรใช้ Expense Adjustment

  • ค่าใช้จ่ายจริงใน Job ถูกบันทึกไปแล้ว แต่ต้องแก้จำนวนเงิน / รายละเอียดหลังจากนั้น
  • Job มี cash advance ที่โอนแล้ว ทำให้แก้ actual expense ฐานตรง ๆ ไม่ได้
  • มีรายการ adjustment เก่าอยู่ใน flow เดียวกัน จึงต้องใช้ revision ผ่าน adjustment ไม่ใช้ overwrite

เงื่อนไขก่อนสร้าง Adjustment

  • ผู้ใช้ต้องมีสิทธิ์ดูแล Job และมีสิทธิ์ expense_adjustment create หรือ submit
  • Job นั้นต้องมี Cash Advance ที่ผ่านขั้นโอนแล้วอย่างน้อย 1 ใบ
  • Adjustment ถูกผูกกับ actual expense และ cash advance ที่เกี่ยวข้อง

Flow ของ Expense Adjustment

Draft ผู้ใช้ร่างรายการและแก้ไขได้
→
Pending CS Supervisor รอหัวหน้า CS อนุมัติ
→
Pending Finance Manager รอผู้จัดการการเงินอนุมัติ
→
Approved effective actual ของ expense ถูกอัปเดตจาก adjustment ที่อนุมัติล่าสุด

ตารางสถานะของ Expense Adjustment

สถานะ ใครทำงานต่อ Action หมายเหตุ
Draft ผู้ขอ แก้ไข, ลบ, ส่งอนุมัติ ส่งแล้วไป CS Supervisor
Pending CS Supervisor หัวหน้า CS Approve, Return to Draft, Reject ผู้ขอเองอนุมัติรายการตัวเองไม่ได้
Pending Finance Manager ผู้จัดการการเงิน Approve, Return to Draft, Reject ถ้ามี assigned user ระบุไว้ ต้องเป็น user คนนั้นหรือ admin
Approved ผู้ดูข้อมูล / ผู้เคลียร์ นำยอด effective actual ไปใช้ใน Settlement ฐาน actual เดิมยังไม่ถูก overwrite ตรง ๆ แต่ระบบดึงค่า effective ล่าสุดมาใช้
Rejected ผู้ขอ แก้ไข, ส่งใหม่, ลบ แก้ได้เฉพาะ draft หรือ rejected เท่านั้น
ความต่างจาก Cash Advance: Expense Adjustment ใช้หลัก “สิทธิ์ + ผู้ที่ถูก assign” ถ้าระบุ approver ไว้ ระบบจะยอมให้เฉพาะ user คนนั้นกดอนุมัติ ยกเว้น admin

8. Expense Control และ Bulk Approval

Expense Control คือ workspace สำหรับคนคุมงานด้านค่าใช้จ่าย ใช้ดู queue งานที่ยังไม่ปิด และช่วยลดการเข้าออกหลายหน้า

สิ่งที่เห็นใน Expense Control

  • ยอดและจำนวน Cash Advance ของวันนี้
  • ยอดและจำนวน Cash Advance ของสัปดาห์นี้
  • ยอดรายการรอโอน
  • ยอดรายการโอนแล้ว / รอเคลียร์
  • รายการ Expense Adjustment ที่ยังเปิดอยู่
  • Settlement modal แบบเร็วจากรายการบางส่วนของวันนี้

กติกาของ open queue

ในมุม Expense Control ระบบจะไม่นับสถานะต่อไปนี้เป็นคิวเปิด:

  • draft
  • settled
  • rejected
  • cancelled

Bulk Approval ใช้เมื่อไร

Bulk CS Approval

เหมาะกับวันไหนที่มีใบเบิกจำนวนมากและผู้ตรวจดูรายละเอียดไว้ก่อนแล้ว

Bulk Finance Verify

ใช้กับรายการที่ account code และ routing ถูกต้องครบแล้ว

Bulk Final Approval

ใช้เฉพาะกรณีที่ต้องการอนุมัติทุก line ของทุกใบที่เลือกพร้อมกัน

แนวปฏิบัติที่แนะนำ: ถ้ารายการใดมีความเสี่ยงต้อง reject บาง line หรือมีรายละเอียดพิเศษ ให้เปิดหน้า detail แล้วอนุมัติรายใบ แทนการใช้ Bulk Final Approval

9. Business Rules สำคัญ

1 ใบเบิก = 1 Job

ปัจจุบันยังไม่รองรับ 1 ใบเบิกที่ผูกหลาย Job ระบบใหม่ที่ทำแล้วคือ 1 ใบเบิกมีหลาย line ได้ แต่ยังอยู่ใน Job เดียว

1 line = 1 ประเภทค่าใช้จ่าย

ไม่ได้หมายถึง 1 ใบเสร็จ ดังนั้นตอน Settlement จึงสามารถแตก 1 line ไปหลาย actual expense / หลายใบเสร็จได้

อนุมัติขั้นสุดท้ายแบบบาง line ได้

แต่ต้องมีอย่างน้อย 1 line ที่ผ่านจริง และ line ที่ไม่ผ่านต้องมีเหตุผลกำกับ

หลังโอน ห้ามแก้ใบเบิก

หลังสถานะเป็น Transferred ต้องแก้ความจริงผ่าน Settlement หรือ Expense Adjustment เท่านั้น

ก่อนโอน ถ้าแก้ข้อมูลสำคัญ ระบบ re-approve ใหม่

ใช้กับช่วง Pending Final Approval และ Pending Accounting ถ้ามีการเปลี่ยนข้อมูลที่มีผลต่อยอดหรือบริบทคำขอ

Account Code สำคัญก่อนการเงินตรวจสอบ

ทุก request line ต้องมี Account Code ก่อนผ่านขั้น Finance Verify

Actual Expense บางครั้งแก้ตรง ๆ ไม่ได้

ถ้ามี cash advance หลังโอน หรือมี adjustment ค้าง/อนุมัติแล้ว รายการฐานจะ lock และต้องแก้ผ่าน revision flow แทน

Settlement ปิดได้เมื่อส่วนต่างเป็นศูนย์

ไม่ว่าจะแก้ด้วย allocation, top-up, return หรือ manual adjustment สุดท้าย remaining variance ต้องเป็น 0

10. คำถามที่พบบ่อย

ถาม: เบิกล่วงหน้าทีเดียว แต่ตอนเคลียร์อยากแตกตามใบเสร็จหลายใบได้ไหม

ได้ในรูปแบบปัจจุบัน ถ้าใช้ 1 ใบเบิกแบบหลาย request line และไปแตก actual expense หลายรายการตอน Settlement โดยแต่ละ line สามารถ allocate ไปหลายค่าใช้จ่ายจริงได้

ถาม: ยอดโอนคืนจะไปอยู่ตรงไหน

อยู่ใน Settlement ที่ระดับ line ผ่านช่อง Return Amount และสรุปยอดรวมที่หน้า Settlement Summary ของใบเบิก

ถาม: ลูกค้ายกเลิกกะทันหันหลังโอนแล้วต้องทำอย่างไร

ไปที่ Settlement บันทึกค่าใช้จริงที่เกิดขึ้นแล้ว ถ้ามีเงินเหลือให้ใส่ Return Amount และเลือก Settlement Reason เป็น Job cancelled

ถาม: หลังส่งอนุมัติแล้วยังแก้ได้ไหม

ก่อนโอนยังแก้ได้บางสถานะ แต่ถ้าแก้ข้อมูลสำคัญในช่วง Pending Final Approval หรือ Pending Accounting ระบบจะเริ่ม approval ใหม่ ส่วนหลังโอนแล้วแก้ใบเบิกไม่ได้ ต้องใช้ Settlement

ถาม: ทำไมเห็นรายการแต่กดอนุมัติไม่ได้

สาเหตุหลักมี 3 แบบ คือ 1) สถานะยังไม่ถึงขั้นนั้น 2) ไม่มี permission ของ stage นั้น 3) สำหรับ Expense Adjustment รายการถูก assign ให้คนอื่น

ถาม: ทำไมปิด Settlement ไม่ได้

ส่วนใหญ่เกิดจาก remaining variance ยังไม่เป็นศูนย์, ยังไม่ได้เลือก Settlement Reason ตอน close หรือยังไม่มีเอกสารแนบอย่างน้อย 1 ไฟล์

ถาม: ทำไมแก้ Actual Expense เดิมไม่ได้

เพราะระบบ lock รายการฐานหลังมี cash advance โอนแล้ว หรือเมื่อมี expense adjustment เปิด/อนุมัติอยู่ ต้องใช้ Expense Adjustment แทน
